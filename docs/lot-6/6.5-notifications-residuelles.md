# Lot 6.5 — Notifications résiduelles

## Objectif

Combler 2 trous fonctionnels qui dégradaient l'UX MIZNAS :
- **6.5.A** Forgot password self-service — un user qui oublie son mdp
  pouvait seulement demander un reset à son admin (coût opérationnel
  + délai). Lot 6.5.A livre le flux standard `/auth/forgot-password`
  + `/auth/reset-password` avec token UUID v4 stocké en hash SHA-256.
- **6.5.B** Notification J-3 délégation — aucun rappel proactif
  n'avertissait le délégant ni le délégataire qu'une délégation
  approchait de sa date de fin. Risque concret : reprise de poste
  avec saisies bloquées en fin de période. Lot 6.5.B livre le cron
  quotidien `0 6 * * *` qui notifie 3 jours avant l'expiration.

L'architecture cron + queue + audit posée par les Lots 6.3 et 6.4
est intégralement réutilisée — pas de nouvelle dépendance npm.

### 6.5.B — Cron rappel J-3 délégation

```
   ┌────────────────────────────────────────┐
   │ Cron @Cron('0 6 * * *')                │
   │ + OnApplicationBootstrap (rattrapage)  │
   └────────────┬───────────────────────────┘
                ▼
   ┌────────────────────────────────────────┐
   │ DelegationsRappelService.notifierJ3()  │
   ├────────────────────────────────────────┤
   │ SELECT id FROM delegations             │
   │  WHERE date_fin = today + 3 jours      │
   │    AND actif = true                    │
   │    AND derniere_notification_j3 IS NULL│
   ├────────────────────────────────────────┤
   │ Pour chaque délégation :               │
   │  ─ envoyerEmail(DELEGANT, delegant)    │
   │     │ opt-out → SUPPRIME + motif       │
   │     │ ok      → EN_ATTENTE + queue     │
   │  ─ envoyerEmail(DELEGATAIRE, deleg.)   │
   │  ─ UPDATE derniere_notification_j3     │
   │  ─ audit DELEGATION_RAPPEL_J3 (1×)     │
   └────────────────────────────────────────┘
```

## Spécifications

### Token reset password

| Caractéristique | Valeur |
|---|---|
| Format clair | UUID v4 (36 chars, `node:crypto.randomUUID()`) |
| Format stocké | Hash SHA-256 hex (64 chars) |
| Validité | 30 minutes |
| Usage | Une seule fois (`utilise=true` après reset) |
| Stockage | Table `password_reset_token` (FK CASCADE vers user) |
| Cleanup | Cron quotidien 03:00, supprime tokens > 30 jours |

### Rate limit forgot-password

| Caractéristique | Valeur |
|---|---|
| Clé | IP uniquement (anti-énumération via 429) |
| Fenêtre | 15 minutes |
| Limite | 3 tentatives |
| Réponse au blocage | 429 + header `Retry-After` + code `LOGIN_RATE_LIMITED` |
| Bypass | `LOGIN_RATE_LIMIT_DISABLED=true` (dev/test only) |

### Cron J-3 délégation

| Caractéristique | Valeur |
|---|---|
| Cron | `0 6 * * *` (06:00 heure serveur) |
| Bootstrap | `OnApplicationBootstrap` (rattrapage si app down) |
| Sélection | `date_fin = CURRENT_DATE + INTERVAL '3 days' AND actif AND derniere_notification_j3 IS NULL` |
| Idempotence | UPDATE `derniere_notification_j3 = now()` après publication |
| Opt-out user | Respecté (filtre `notificationsEmailActives` + `notificationsEmailTypes`) → `email_log SUPPRIME` |
| Audit | `DELEGATION_RAPPEL_J3` (1 entrée par délégation, peu importe le nombre de destinataires) |

## Sécurité — invariants validés

| Invariant | Vérification | Test |
|---|---|---|
| Token clair JAMAIS persisté en base | hash SHA-256 stocké dans `password_reset_token.token` | `forgot-password.e2e-spec.ts` (assert SQL `MATCH /^[a-f0-9]{64}$/`) |
| Token clair JAMAIS dans `email_log.payload` | INSERT email_log avec payload `{ raison, expiration_minutes }` ; le clair transit via `EmailJobData.secrets` (Redis éphémère) | `forgot-password.e2e-spec.ts` |
| Token clair JAMAIS dans `audit_log.payload_apres` | audit `DEMANDE_RESET_MDP_USER` avec `payloadApres = { email, expiration_minutes }` | `forgot-password.e2e-spec.ts` |
| Réponse forgot-password EXACTEMENT identique pour email connu/inconnu | `reponseUniformeForgot()` retourne le même objet | `password-reset.service.spec.ts` + `forgot-password.e2e-spec.ts` |
| Token usable une seule fois | `utilise=true` après reset, 2ème POST → `INVALID_TOKEN` | `forgot-password.e2e-spec.ts` |
| Token expiré rejeté distinctement | `date_expiration < now()` → `410 EXPIRED_TOKEN` (vs `400 INVALID_TOKEN`) | `forgot-password.e2e-spec.ts` |
| Rate limit forgot-password par IP | 3/15min, 4ème = 429 | `forgot-password.e2e-spec.ts` |

## Dette tracée

- **Pas de pre-check du token** au chargement de
  `/reset-password?token=XYZ` (décision actée Lot 6.5.A) — l'erreur
  remonte au submit. UX courante (Google, GitHub) ; à reconsidérer
  Lot 6.7 si retour utilisateur négatif.
- **Anti-chaînage strict des emails J-3** : si le délégataire est
  lui-même délégant d'une autre délégation expirant J-3, il reçoit
  2 emails distincts. Acceptable V1.
- **Bandeau in-app J-3 délégation** non livré (mandat reporté
  Lot 6.7) — la notification est uniquement par email.
- **`derniere_notification_j3` jamais reset** : si un admin
  réactive une délégation après expiration, le flag reste posé. À
  reconsidérer Lot 7+ si besoin (cas marginal).

## Variables d'environnement

Aucune nouvelle variable. Réutilisation de :
- `APP_BASE_URL` (lien reset password dans le mail)
- `LOGIN_RATE_LIMIT_DISABLED` (bypass forgot-password rate limit)
- `EMAIL_DRY_RUN` (mode dry-run du module notifications)
- `MDP_DUREE_VALIDITE_JOURS` (Lot 6.4.A — applique aussi à
  `executerReset` pour positionner `date_expiration_mdp`)

## Codes audit ajoutés (migration 220)

Convention FR métier (cohérent avec `RESET_PASSWORD_USER` du Lot
Administration).

| Code | Quand |
|---|---|
| `DEMANDE_RESET_MDP_USER` | Email connu actif, lien généré + envoyé |
| `DEMANDE_RESET_MDP_INCONNU` | Email inconnu/inactif, AUCUNE I/O DB |
| `RESET_MDP_USER_VALIDE` | Token validé + mdp changé |
| `NETTOYAGE_RESET_TOKENS` | Cron cleanup quotidien (count > 0 uniquement) |
| `DELEGATION_RAPPEL_J3` | 1 entrée par délégation notifiée par cron J-3 |
