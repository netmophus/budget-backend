# Lot 6.3 — Queue BullMQ + Redis pour emails async

## Objectif

Découpler l'envoi des emails de l'API NestJS pour rendre les
controllers résilients à un SMTP lent ou en panne. Avant 6.3,
`POST /budget/version/:id/soumettre` attendait jusqu'à 14s
(retries SMTP synchrones 1s/3s/10s) avant de répondre. Après :
réponse immédiate, l'email part en queue et le worker l'envoie
en arrière-plan.

## Architecture

```
┌──────────────────┐         ┌──────────────────┐
│   API NestJS     │         │     Worker       │
│  (controllers)   │         │  (in-process)    │
└────────┬─────────┘         └────────▲─────────┘
         │                            │
         │ 1. INSERT email_log        │ 4. consume job
         │    statut=EN_ATTENTE       │
         │                            │
         │ 2. publier(emailLogId)     │ 5. UPDATE statut=EN_COURS
         │                            │
         ▼                            │ 6. SMTP (nodemailer)
┌─────────────────────────────────────┴──────────┐
│            Queue BullMQ 'emails' (Redis)       │
│  attempts=3, backoff exponential 2s/4s/8s      │
│  removeOnComplete=100, removeOnFail=1000       │
└────────────────────────────────────────────────┘
                         │
                         │ 7a. succès → UPDATE ENVOYE + envoyeLe
                         │ 7b. échec → BullMQ retry
                         │            ou ECHEC après dernier retry
                         ▼
                  ┌──────────────┐
                  │  email_log   │
                  │  (Postgres)  │
                  └──────────────┘
```

### Composants

- **`EmailQueueProducer`** (`src/notifications/email-queue.producer.ts`)
  Publie les jobs avec options par défaut (attempts=3, backoff
  exponential 2s/4s/8s). Expose aussi `pingRedis()` (healthcheck) et
  `getQueueStats()` (admin endpoint).

- **`EmailWorker`** (`src/notifications/email.worker.ts`)
  `@Processor('emails')` qui consume la queue. Délègue à
  `NotificationsService.traiterJob()`. Détecte la dernière tentative
  via `attemptsMade + 1 >= options.attempts` et appelle
  `marquerEchecDefinitif()` pour basculer la ligne en ECHEC.

- **`NotificationsService.traiterJob(emailLogId, attemptsMade)`**
  Charge `email_log`, bascule en EN_COURS, rend le template Handlebars,
  tente le SMTP via nodemailer (1 seule tentative — c'est BullMQ qui
  gère les retries). Succès → ENVOYE. Échec → throw (laisse BullMQ
  décider de retry ou de mettre en failed).

### Statuts `email_log` (Lot 6.3)

| Statut       | Sens                                                    |
| ------------ | ------------------------------------------------------- |
| `EN_ATTENTE` | Job publié dans la queue, en attente du worker          |
| `EN_COURS`   | Worker a pris le job, envoi SMTP en cours (NEW Lot 6.3) |
| `ENVOYE`     | Envoi SMTP réussi                                       |
| `ECHEC`      | Toutes les retries BullMQ ont échoué (terminal)         |
| `SUPPRIME`   | Email non envoyé volontairement (dry-run / pref user)   |

Migration `1779200000170-AjoutStatutEnCoursEmailLog` ajoute
`EN_COURS` au CHECK constraint `chk_email_log_statut`.

## Configuration

### Variables d'env

| Variable          | Default     | Rôle                                  |
| ----------------- | ----------- | ------------------------------------- |
| `REDIS_HOST`      | `localhost` | Host Redis                            |
| `REDIS_PORT`      | `6379`      | Port Redis                            |
| `REDIS_PASSWORD`  | (vide)      | Auth optionnelle (prod managé)        |
| `EMAIL_DRY_RUN`   | `true`      | Si `true`, statut SUPPRIME sans queue |
| `SMTP_HOST/PORT`  | localhost:1025 | nodemailer transporter             |

### Redis local (dev)

```bash
docker compose -f docker-compose.dev.yml up -d
# Stop : docker compose -f docker-compose.dev.yml down
```

L'image `redis:7-alpine` tourne sur `:6379` avec un volume persistant
pour conserver les jobs entre redémarrages.

### Redis prod

**Ne pas utiliser le Docker Compose local en production.** Brancher un
Redis managé via les variables d'env :

| Provider        | Host pattern                                 |
| --------------- | -------------------------------------------- |
| AWS ElastiCache | `<cluster-id>.<az>.cache.amazonaws.com`      |
| GCP Memorystore | `<instance-id>.<region>.c.<project>.internal`|
| Upstash         | `<endpoint>.upstash.io`                      |
| Render Redis    | `<service>.internal`                         |

Le password (TLS / AUTH) passe par `REDIS_PASSWORD`. La connexion est
mémoïsée par `BullModule.forRootAsync` au boot.

### E2E

- `test/e2e/emails.e2e-spec.ts` — Postgres + Redis via testcontainers,
  flux complet :
  - **SENT** : insertion email_log + queue.add → SMTP mocké réussi →
    statut ENVOYE.
  - **FAILED** : insertion + queue.add (attempts=1) → SMTP mocké
    rejette → statut ECHEC + dernierMessageErreur tracé.

Le setup global (`test/e2e/setup-global.ts`) démarre Postgres et Redis
en parallèle (~16s contre ~6s avec Postgres seul). En CI, le runner
`ubuntu-latest` a Docker préinstallé donc testcontainers fonctionne
nativement (cf. job `test-e2e` dans `.github/workflows/ci.yml`,
inchangé depuis le Lot 6.2.A).

## Healthcheck

`GET /api/v1/health` (public, Lot 6.3.C — refactor) :

```jsonc
{
  "status": "ok",                 // ou "degraded" si Redis down
  "app":      { "status": "up" },
  "database": { "status": "up" },
  "redis":    { "status": "up" }  // ou "down" + "message"
}
```

- DB down → HTTP 503 (l'app n'est pas utilisable sans la base).
- Redis down → HTTP 200 + `status: "degraded"` + `redis: {status: "down"}`.
  Décision produit : MIZNAS reste utilisable même sans emails. Les
  monitorings externes peuvent alerter spécifiquement sur
  `redis.status` sans crasher tout le supervision.

## Monitoring

### Endpoint admin queue stats

`GET /api/v1/admin/email-log/queue/stats` (permission `USER.GERER`) :

```json
{
  "waiting": 0,
  "active": 0,
  "completed": 142,
  "failed": 3,
  "delayed": 0
}
```

Source : `queue.getJobCounts()` BullMQ. Permet de détecter :

- `waiting` élevé persistant → worker arrêté ou Redis lent
- `failed` croissant → SMTP en panne durable, vérifier `email_log` +
  alerter ops
- `delayed` non nul → backoff en cours (jobs replanifiés après échec)

### Debug via redis-cli

```bash
# Connexion (dev local)
redis-cli -h localhost -p 6379

# Lister les jobs en attente
LRANGE bull:emails:waiting 0 -1

# Lister les jobs failed
LRANGE bull:emails:failed 0 -1

# Détail d'un job (par id)
HGETALL bull:emails:42

# Compteurs globaux
HGETALL bull:emails:meta
```

En prod : `redis-cli -h <managed-host> -a <password> --tls`.

## Dette tracée

- **Worker in-process V1.** Le `@Processor` tourne dans le process
  NestJS de l'API. Avantages : simplicité, déploiement unique. Limite :
  pas de scale indépendant (si l'API a un pic CPU, le worker ralentit).
  À isoler en process dédié au **Lot 7+** (orchestration / scaling).

- **Pas d'UI bull-board.** Pour cette V1, le monitoring se fait via
  l'endpoint `/admin/email-log/queue/stats` + `redis-cli` direct.
  L'UI bull-board (interface web BullMQ) sera évaluée si le besoin
  émerge — pour l'instant, l'admin a déjà la table `email_log` côté
  UI (Lot 4.3) qui couvre 95% des besoins.

- **Pas de code audit `EMAIL_ECHEC_DEFINITIF`.** La trace de l'échec
  est dans `email_log` (statut=ECHEC + dernierMessageErreur) qui est
  l'audit du domaine notifications. Ajouter un code audit demanderait
  une migration + update du type union `TypeAction` (cf. dette
  existante "aligner code audit migration ↔ TypeAction"). Réouvrable
  si besoin émerge côté audit BCEAO.
