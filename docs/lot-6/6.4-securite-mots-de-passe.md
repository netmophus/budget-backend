# Lot 6.4 — Sécurisation des mots de passe

## Objectif

Aligner MIZNAS avec les exigences de sécurité bancaires (BCEAO, ISO
27001) pour la gestion des mots de passe. Le Lot 6.4 livre 3 paliers
indépendants mais cohérents :

- **6.4.A** — Politique mot de passe centralisée, expiration 90 jours,
  endpoint self-service de changement, guard global qui bloque toute
  navigation tant que le mdp doit être changé.
- **6.4.B** — Rate limiting sur `POST /auth/login` pour prévenir le
  brute-force (5 tentatives/60s par IP, 5 tentatives/15min par email).
- **6.4.C** — Reset password admin asynchrone via queue BullMQ, le
  mot de passe en clair n'est plus jamais affiché à l'admin (envoyé
  par email à l'utilisateur, transit via Redis éphémère). Frontend
  bloquant `/change-mdp` qui force le changement avant tout autre
  accès.

## Architecture

### Politique mot de passe partagée

Source unique de vérité : `src/auth/password-policy.ts`.

```
validatePasswordPolicy(mdp: string): { ok: boolean; erreurs: string[] }
@MotDePasseValide() decorator (class-validator) pour DTO
genererMotDePasseTemporaire(longueur = 32): string  // policy-conforme
```

Règles :
- Longueur ≥ 12 caractères
- ≥ 1 majuscule, ≥ 1 minuscule, ≥ 1 chiffre, ≥ 1 caractère spécial

`genererMotDePasseTemporaire` pioche 4 caractères (1 par catégorie),
complète avec des bytes random `node:crypto.randomBytes`, puis
mélange Fisher-Yates. Garantie : satisfait toujours la policy.

### Flux d'expiration et flags JWT (palier A)

```
Migration 180 :
  ALTER user ADD date_expiration_mdp TIMESTAMP NULL
  ALTER user ADD doit_changer_mdp BOOLEAN DEFAULT false

Migration 190 :
  Codes audit PASSWORD_CHANGED + LOGIN_RATE_LIMITED ajoutés à la
  contrainte CHECK de audit_log.type_action.
```

À chaque login, `AuthService.login()` calcule :
- `mdpExpire = date_expiration_mdp < now()` (90 jours par défaut,
  configurable via `MDP_DUREE_VALIDITE_JOURS`)
- `doitChangerMdp = u.doit_changer_mdp` (flag DB direct)

Réponse `POST /auth/login` :
```json
{
  "accessToken": "...",
  "refreshToken": "...",
  "user": { "id", "email", "nom", "prenom" },
  "expiresIn": 900,
  "mdpExpire": false,
  "doitChangerMdp": true
}
```

Le JWT payload encode aussi ces flags (`mdpExpire?: boolean`,
`dcm?: boolean` — abrégé pour limiter la taille du token).

### `PasswordExpiredGuard` (palier A)

Guard global enregistré comme `APP_GUARD`. À chaque requête
authentifiée, vérifie le payload JWT :
- Si `mdpExpire || dcm` → throw `ForbiddenException` avec
  `{ statusCode: 403, code: 'MDP_EXPIRE' | 'MDP_TEMPORAIRE',
   message: '...' }`
- Bypass via `@Public()` (déjà existant) ou `@AllowExpiredPassword()`
  (nouveau, palier A).

Routes whitelistées via `@AllowExpiredPassword()` :
- `PATCH /me/password` (le user doit pouvoir changer son mdp)
- `GET /auth/me`, `POST /auth/logout` (info session + déconnexion)

### Endpoint self-service `PATCH /me/password` (palier A)

```
Body : { ancienMdp, nouveauMdp } (DTO `ChangerMdpDto`, validations
       class-validator + @MotDePasseValide())
Auth : Bearer (JWT), Guard `JwtAuthGuard`
       + @AllowExpiredPassword() (bypass PasswordExpiredGuard)

Service : AuthService.changerMdp(userId, ancien, nouveau, ip, ua)
  1. bcrypt.compare(ancien, u.mot_de_passe_hash) → 401 si KO
  2. bcrypt.hash(nouveau, BCRYPT_ROUNDS=12)
  3. UPDATE user :
     - mot_de_passe_hash = nouveau hash
     - date_expiration_mdp = now() + 90j
     - doit_changer_mdp = false
  4. Audit `PASSWORD_CHANGED`
  5. issueTokens(user, flags=undefined) — nouveaux tokens SANS flags

Réponse : { accessToken, refreshToken, user, expiresIn }
```

### Rate limiting `POST /auth/login` (palier B)

Service `LoginRateLimiterService` (in-memory, mono-instance V1) :
- `parIp` : Map<ip, timestamps[]> — fenêtre 60s, limite 5
- `parEmail` : Map<email_lowercase, timestamps[]> — fenêtre 15min,
  limite 5

Désactivable via env var `LOGIN_RATE_LIMIT_DISABLED=true` (utilisé
en e2e backend + Playwright local).

Guard `LoginRateLimitGuard` appliqué uniquement sur `POST /auth/login`
(via `@UseGuards`). Ne consomme PAS le compteur quand bloqué (sinon
la fenêtre ne décroît jamais). Si bloqué :
- audit `LOGIN_RATE_LIMITED` avec `{ email tenté, ip, motif: 'IP'|'EMAIL' }`
- header HTTP `Retry-After: <secondes>`
- throw `429` avec `{ code: 'LOGIN_RATE_LIMITED', message }`

### Reset password admin async (palier C.1)

**Breaking change DTO** : `ResetPasswordResponseDto = { success, message }`.
Le mot de passe temporaire n'est plus dans la réponse API.

```
Service : UsersAdminService.resetPassword(targetId, currentUser)
  1. Génère mdp temporaire 32 chars policy-conforme
     via genererMotDePasseTemporaire(32)
  2. UPDATE user :
     - mot_de_passe_hash = bcrypt.hash(mdp, 12)
     - doit_changer_mdp = true
     - date_expiration_mdp = now() + 7j (validité courte du temp)
  3. INSERT email_log statut=EN_ATTENTE
     - type='RESET_PASSWORD_ADMIN'
     - destinataire = u.email
     - payload = { nom, prenom, expirationISO, ... }  // PAS le mdp
     - hors transaction principale (commit avant publish queue)
  4. EmailQueueProducer.publier(emailLogId, secrets={ mdpTemporaire: mdp })
     - publié HORS transaction
     - secrets transitent uniquement dans Redis (job data),
       jamais persistés dans email_log.payload
  5. Audit `RESET_PASSWORD_USER` avec payloadApres = { email,
     doitChangerMdp: true } (PAS le mdp)
  6. Retourne { success: true, message: 'Email envoyé à <email>' }
```

**Architecture émergée** : `EmailQueueModule`. Le palier C.1 a
révélé que `NotificationsModule` exposait son `EmailWorker`
transitivement, ce qui faisait crash les e2e backend pg-mem
(BullExplorer scanne les `@Processor` au boot, exige Redis).
Refactor : extraction du `EmailQueueProducer` (+ `BullModule.registerQueue`)
dans `EmailQueueModule`. `NotificationsModule` importe ce module +
ajoute `EmailWorker` localement. `UsersModule` importe seulement
`EmailQueueModule` (pas de Worker propagé).

### Force change password — endpoint admin (palier C.2)

```
@Post(':id/forcer-changement-mdp')
@RequirePermissions('USER.GERER')
forcerChangementMdp(id, currentUser)
  1. UPDATE user SET doit_changer_mdp = true
  2. NE TOUCHE PAS au mot de passe existant
  3. Audit `RESET_PASSWORD_USER` avec
     payloadApres = { ..., operation: 'forcer-changement-mdp' }
  4. Retourne UserResponseDto
```

Cas d'usage : suspicion de compromission, le support force le
changement mais conserve le mdp courant (le user garde sa session
active mais ne peut plus rien faire avant changement).

### Frontend `/change-mdp` (palier C.2)

- `auth-store.ts` (Zustand persist) : flags `mdpExpire`, `doitChangerMdp`
  hydratés depuis le login. `useDoitChangerMdp()` retourne
  `mdpExpire || doitChangerMdp`. Méthode `changerMdp(ancien, nouveau)`
  appelle `PATCH /me/password` et remplace les tokens (sans flags).
- `ProtectedRoute` : redirige vers `/change-mdp` si flag posé ET
  pathname ≠ `/change-mdp`.
- `ForceChangePasswordPage` (route `/change-mdp`, hors `AuthLayout`) :
  formulaire 3 champs avec validation zod policy-conforme. Si flags
  absents (arrivée intempestive) → `<Navigate to="/dashboard" />`.
  Pattern déclaratif `<Navigate />` (pas `navigate()` impératif —
  cf. dette tracée dans le commit `fix(routing)`).

## Sécurité — invariants validés

| Invariant | Vérification | Test |
|---|---|---|
| Mot de passe en clair jamais retourné par l'API | `ResetPasswordResponseDto = { success, message }` | `users-admin.service.spec.ts` |
| Mot de passe en clair jamais persisté en `email_log.payload` | INSERT email_log se fait sans mdp ; payload contient nom/prenom/expirationISO | `reset-password-admin.e2e-spec.ts` |
| Mot de passe en clair jamais persisté en `audit_log` | Audit `RESET_PASSWORD_USER` avec `payloadApres = { email, doitChangerMdp }` | spec users-admin |
| Mot de passe temporaire transit via Redis éphémère | `EmailJobData.secrets` injecté dans Handlebars au moment de l'envoi par le worker, pas avant | `email.worker.spec.ts` |
| Force changement à la prochaine connexion | `doit_changer_mdp = true` après reset → JWT `dcm: true` → `PasswordExpiredGuard` 403 | `password.e2e-spec.ts` |
| Brute force login limité | 5/60s IP + 5/15min email, audit `LOGIN_RATE_LIMITED` | `rate-limit.e2e-spec.ts` |
| Session non révoquée par `forcer-changement-mdp` | Le hash mdp n'est pas changé, le flag est posé | spec users-admin |

## Dette tracée

- **Storage rate limit in-memory** : non distribué, OK pour V1
  mono-instance. Migration vers Redis storage prévue Lot 7+ (BullMQ
  Redis du Lot 6.3 déjà disponible).
- **`LOGIN_RATE_LIMIT_DISABLED=true` dans `.env` dev local** : à
  retirer / forcer `false` en pré-prod et prod (commenté dans
  `.env.example`).
- **`BandeauMdpExpire` (frontend)** : alerte UI quand le mdp
  expire dans X jours. Reporté Lot 6.7 (UX résiduel).

## Variables d'environnement

| Var | Type | Défaut | Description |
|---|---|---|---|
| `MDP_DUREE_VALIDITE_JOURS` | number | `90` | Durée de validité du mdp (palier A) |
| `LOGIN_RATE_LIMIT_DISABLED` | boolean | `false` | Désactive le rate limit login (dev/test only) |

Cf. `.env.example` pour la documentation à jour.

## Audit codes ajoutés (migration 190)

- `PASSWORD_CHANGED` — émis par `PATCH /me/password` (self-service)
  et `forcer-changement-mdp` admin
- `LOGIN_RATE_LIMITED` — émis par `LoginRateLimitGuard` quand le
  seuil est franchi
- `RESET_PASSWORD_USER` — étendu : précise l'opération via
  `payloadApres.operation` (`reset-password` ou
  `forcer-changement-mdp`)
