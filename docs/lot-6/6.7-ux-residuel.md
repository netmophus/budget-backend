## Sous-lots livrés

### 6.7.1 — BandeauMdpExpire (Backend + Frontend)

#### Décision technique clé

Le diagnostic a révélé que le booléen `mdpExpire` existant
(Lot 6.4) signifie « **déjà expiré** » (bloque l'accès via
`PasswordExpiredGuard` backend + `ProtectedRoute` frontend),
**pas J-7 avant expiration**. Un bandeau conditionné sur
`mdpExpire` ne se serait jamais affiché dans l'app (l'utilisateur
est forcé sur `/change-mdp` avant d'atteindre `AuthLayout`).

→ **Option (a') validée** : ajout d'un **booléen distinct**
`mdpExpireProchainement` dans `LoginResponse`, mutuellement
exclusif avec `mdpExpire` (vrai uniquement si
`dateExpirationMdp ∈ ]now, now+7j[`).

#### Backend — commit `a9f6cfc`

- `auth.service.ts` : calcul `mdpExpireProchainement` dans
  `login()` + retour dans `LoginResult` (interface enrichie) +
  retour `false` dans `changerMdp()` (date renouvelée à
  ~90 jours après changement)
- `auth.controller.ts` : champ exposé dans l'interface `LoginResponse`
- 4 tests Jest régression : cas J+5 (true), J+10 (false), déjà
  expiré (false, exclusion mutuelle), `dateExpirationMdp` null (false)
- Mock `auth.controller.spec` synchronisé avec le nouveau champ

#### Frontend — commit `44cf1c6`

- Types `LoginResponse` + `ChangerMdpResponse` (`src/lib/api/types.ts`)
  enrichis de `mdpExpireProchainement`
- Store Zustand `auth-store.ts` persiste `mdpExpireProchainement`
  (résiste au refresh page) + hook `useMdpExpireProchainement()`
- Composant `BandeauMdpExpire.tsx` (orange, alerte modérée) inséré
  globalement dans `AuthLayout.tsx` au-dessus de `<Outlet />`
- **Extension `ForceChangePasswordPage` (Lot 6.4.C.2)** pour accepter
  le cas J-7 :
  - Condition d'accès : `!mdpExpire && !doitChangerMdp && !mdpExpireProchainement` → redirect dashboard
  - Texte `raison` : 3 cas distincts (temporaire / expiré / J-7)
  - **Bouton "Plus tard" VISIBLE uniquement en cas J-7 pur** (`mdpExpireProchainement && !mdpExpire && !doitChangerMdp`) — caché si `mdpExpire` ou `doitChangerMdp` pour éviter le contournement du `PasswordExpiredGuard` backend
  - Footer texte adapté : « recommandé » en J-7, « obligatoire » sinon
- 9 tests Vitest régression (3 BandeauMdpExpire + 6 ForceChangePasswordPage incluant les assertions sécurité « Plus tard caché »)
- Cleanup annexe : retrait `eslint-disable` inutile dans `src/types/jsx.d.ts:17`

### 6.7.3 — Édition reforecast inline (Frontend)

#### Décision technique clé

L'audit préalable a révélé que **l'édition reforecast existe
déjà** via redirection — le mandat initial était basé sur une
hypothèse fausse :

- `ReforecastGrille.tsx:173-183` affiche un bouton « Éditer dans
  la saisie budgétaire » si éditable
- Clic → redirige vers `/budget/saisie?versionId={reforecastId}&scenarioId=...`
- `BudgetSaisieService.saveGrilleSaisie` gère déjà l'édition car
  `fait_budget` est partagé entre budget et reforecast (même table)

→ **Option O1 validée** (drop la fausse création) + **mini-amélioration UX (~15 min)** au lieu d'une implémentation O2-O4 qui aurait dupliqué 95% de `SaisieBudgetairePage` pour un gain marginal.

#### Commit `8d8e839`

- **Mini-fix drift** : `TypeVersion` frontend (`src/lib/api/versions.ts`)
  étendu avec `'reforecast'` (le backend `DimVersion` l'a depuis
  Lot 5.3, le frontend ne l'avait jamais aligné — bug silencieux
  probable dans `VersionsPage` et `VersionFormDrawer`, tracé pour
  Lot 7+). Interface `Version` enrichie des champs reforecast
  optionnels (`trimestreConsolide`, `anneeConsolide`,
  `methodeExtrapolation`, `statutPublication`)
- `ReforecastGrille.tsx` : bouton renommé **« Éditer ce reforecast »**
  + `<Tooltip>` explicatif au survol : « La saisie reforecast
  utilise la même grille que le budget. Vous serez redirigé vers
  la page de saisie filtrée sur ce reforecast. »
- `SaisieBudgetairePage.tsx` : **bandeau bleu informatif** affiché
  conditionnellement quand `versionComplete?.typeVersion === 'reforecast'` :
  « Vous éditez un reforecast T{trim} {annee}. Les modifications

- 2 tests Vitest régression (bandeau visible si reforecast / caché

dans les entités backend. Toutes les occurrences trouvées sont

- `src/budget/services/perimetre.service.ts:287,331` — paramètre de méthode
- `src/budget/services/perimetre.service.spec.ts:73,197` — variables Map de seed
- `src/budget/services/budget-saisie.service.spec.ts:237,388,458` — variables Map de seed

Les colonnes FK réelles s'appellent **`fk_structure`** (naming

Mandat invalide en l'état. Si un refactor périmètre/structure est
nécessaire à terme, il sera traité **Lot 7+ avec scope clair**
(pas une « suppression » mais une révision du modèle de
permissions).

## Disciplines méthodologiques appliquées

### 1. Diagnostic préalable systématique avant chaque tâche

Discipline acquise au Lot 6.6 (« relire avant fix »). Sur Lot 6.7,
chaque sous-tâche a démarré par une phase de diagnostic d'audit
(~30 min chacune) avant toute modification de code. Bénéfice
quantifiable :

- **6.7.1** : 2 incohérences sémantiques détectées avant code
  (mdpExpire = déjà expiré, `/change-mdp` redirige si pas
  bloqué) → option (a') backend ajustée en conséquence (ajout
  flag distinct au lieu de réutiliser le booléen existant)
- **6.7.3** : audit a montré que l'édition reforecast existe déjà
  via redirect → drop de l'option O2 (sub-onglet saisie embarqué)
  qui aurait coûté 3-4h pour duplication inutile. Économie nette.
- **6.7.4** : audit a démontré que la colonne ciblée n'existe pas
  → drop complet de la tâche.

### 2. Validation user systématique entre paliers

Pour chaque sous-tâche : (a) diagnostic → user valide stratégie
→ (b) implémentation → user valide visuellement → (c) palier
suivant. Aucun palier livré « en bloc » sans visualisation
utilisateur — discipline du Lot 6.6 [feedback_cadence_points_etape]
appliquée.

### 3. Test visuel manuel obligatoire après chaque tâche

Chaque sous-lot a fait l'objet d'un test visuel manuel sur le
dev server avant validation. Les rapports d'implémentation ont
systématiquement séparé « ce que les tests Vitest valident »
de « ce qui nécessite un test visuel humain » (transparence
acquise du Lot 6.6 [feedback CLAUDE.md UI testing]).

### 4. Sécurité préservée par tests d'exclusion

Sous-lot 6.7.1 — bouton « Plus tard » sur `ForceChangePasswordPage`
a des tests d'**absence** explicites :
- `bouton "Plus tard" CACHÉ si mdpExpire=true` (sécurité)
- `bouton "Plus tard" CACHÉ si doitChangerMdp=true` (sécurité)

→ Garantit qu'aucun contournement du `PasswordExpiredGuard`
backend n'est introduit accidentellement à terme.

## Dette tracée pour Lot 7+

### Refactor

- **`SaisiePanel` factorisé budget + reforecast** (~3-4h) :
  extraire `SaisieBudgetairePage` + `useGrilleSaisie` en un
  composant `<SaisiePanel/>` consommable depuis `ReforecastDetailPage`
  (sub-onglet « Saisie »). Permet la vraie édition inline sans
  duplication.

### Bugs latents identifiés

- **Origine REALISE/MANUEL mensongère après édition manuelle dans
  reforecast** : le badge Origine est dérivé côté backend
  `getComparaison()` selon `trimLigne <= trim` (REALISE) vs
  `methode === MANUELLE` (MANUEL) vs sinon EXTRAPOLATION. Si
  l'utilisateur édite manuellement une cellule en T1 (consolidé),
  le badge « REALISE » persiste à tort. Pour fixer : ajouter
  `est_edite_manuellement` dans `fait_budget` OU bloquer
  l'édition des mois consolidés. Pas critique car mois consolidés
  ne devraient pas être édités, mais à corriger pour cohérence.
- **TypeVersion drift backend/frontend** : le frontend n'incluait
  pas `'reforecast'` dans son enum `TypeVersion` (corrigé en Lot 6.7.3).
  Probable cause d'affichages génériques pour les reforecasts dans
  `VersionsPage` et `VersionFormDrawer` (badges/libellés default
  via fallback). À investiguer si visuellement gênant.

### Concurrence

- **Optimistic locking** : non-implémenté ni pour budget ni pour
  reforecast (dernier qui save écrase). Pattern accepté pour MVP
  pilote BSIC. À revisiter Lot 7+ si retours utilisateurs.
