# Lot 4.3 — Notifications email

> Statut : **livré** (2026-05-07) — branche `lot-4/4.3-notifications`

## 1. Contexte et décisions produit

À chaque action métier sensible (workflow budgétaire,
délégations, affectations multi-périmètres), MIZNAS notifie les
parties prenantes par email avec :
- traçabilité complète (table `email_log`),
- préférences utilisateur (opt-out global ou par type),
- mode `dry-run` pour démos / CI sans SMTP.

Décisions produit (figées avant Lot 4.3) :

| Décision | Choix retenu | Conséquence |
|----------|--------------|-------------|
| **D3** — événements email | 8 événements câblés (E1 → E5, E7, E8, E9). E6 (rappel délégation J-3) **reporté Lot 6**. | Pas d'orchestration cron complémentaire ici. |
| **D3-bis** — file d'attente | retry **synchrone** simple 3 tentatives × backoff 1s/3s/10s | BullMQ + Redis reportés Lot 6 |
| **D3-ter** — dry-run | `EMAIL_DRY_RUN=true` par défaut en dev | Aucun envoi SMTP réel, mais ligne `email_log` avec `statut='SUPPRIME'` créée. |
| **D3-quater** — modèle préférences | **2 colonnes sur `"user"`** plutôt qu'une table dédiée | 1 user = 1 jeu de prefs ; pas de variantes contextuelles ni d'historisation requise |

### 2.2 Préférences utilisateur (extension `"user"`)

| Colonne | Type | Défaut | Notes |
|---------|------|--------|-------|
| `notifications_email_actives` | boolean NOT NULL | `true` | toggle global |
| `notifications_email_types` | text[] NULL | `NULL` | NULL = tous types acceptés ; sinon liste blanche stricte |

Justification 2 colonnes plutôt qu'une table dédiée :
- 1 user = 1 jeu de préférences (pas de variantes par contexte).
- Pas d'historisation des changements requise (audit déjà couvert
  via `@Auditable` sur `User`).
- Sur-architecture pour 2 colonnes (boolean + array).
- Migration aisée vers une table dédiée si besoin futur (langue,
  thème, etc.).

### 3.1 Filtrage par préférences (toujours appliqué)

Pour chaque destinataire candidat :
- Si `notifications_email_actives=false` → ligne `email_log` créée
  avec `statut='SUPPRIME'`, `payload._motifSuppression='PREF_TOGGLE_GLOBAL_OFF'`.
- Si `notifications_email_types` non NULL et l'événement n'est pas
  dans la liste → ligne `SUPPRIME` avec
  `_motifSuppression='PREF_TYPE_NON_SOUSCRIT'`.

La trace est **systématique** (audit BCEAO).

## 4. Architecture

```
Service métier (VersionWorkflow / Delegations / UserPerimetre)
        │
        │ events.emit('budget.submitted', payload)
        ▼
@nestjs/event-emitter EventEmitter2 (singleton enregistré
                                     dans app.module.ts)
        │
        │ @OnEvent('budget.submitted') (async)
        ▼
NotificationsListeners.onBudgetSubmitted
        │
        │ resoudreDestinataires() + envoyer(...)
        ▼
NotificationsService
        │
        ├──> Filtrage préférences
        ├──> Mode dry-run check (EMAIL_DRY_RUN)
        ├──> Rendu Handlebars (template + layout)
        ├──> nodemailer.sendMail (3 tentatives × 1s/3s/10s)
        ▼
email_log (statut ENVOYE / ECHEC / SUPPRIME)
```

Couplage faible : les services métier émettent **et oublient**.
Un échec de notification ne remonte JAMAIS vers l'action métier
déjà committée (try/catch dans le listener).

## 5. Configuration

`.env` (cf. `.env.example`) :

```bash

SMTP_PORT=1025
SMTP_USER=
SMTP_PASS=
SMTP_FROM=miznas@bsic.local
EMAIL_DRY_RUN=true      # défaut dev
APP_BASE_URL=http://localhost:5173
```

### 5.1 Mailhog en dev (recommandé)

```bash
docker run -d --name mailhog -p 1025:1025 -p 8025:8025 mailhog/mailhog
```

Puis dans `.env` :
- `EMAIL_DRY_RUN=false`
- `SMTP_HOST=localhost` `SMTP_PORT=1025`

UI Mailhog sur http://localhost:8025 — captures de tous les emails
sans envoi réel.

### 5.2 Production

SMTP institutionnel BSIC avec STARTTLS + credentials dans le
secret manager. **Ne jamais committer `SMTP_PASS`**. Le service
nodemailer ne logge pas les credentials (à vérifier avant chaque
release majeure).

## 10. Dette résiduelle / Lot 6

- E6 rappel délégation J-3 : nécessite un cron dédié + suivi
  d'envois (anti-doublon par jour).
- Migration retry vers BullMQ + Redis : permettra de :
  - retirer le `setTimeout` synchrone qui peut bloquer la requête
    en cas d'échec SMTP transitoire,
  - persister les jobs en cas de crash applicatif,
  - paralléliser les envois en masse (publication d'une version
    avec 50 destinataires).
- Templates internationalisation (i18n) — actuellement français
  uniquement, à prévoir si MIZNAS est déployée hors UEMOA.
- Page admin : prévisualisation HTML d'un email (rendre le
  template avec son payload pour vérifier le rendu, sans renvoyer).
- Cookie de désinscription en 1 clic (lien dans le pied de page) —
  alternative au passage par `/me/preferences`.
