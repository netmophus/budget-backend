# Lot 4.1 — Multi-périmètres user / périmètre budgétaire

> Statut : **livré** (2026-05-06) — branche `lot-4/4.1-multi-perimetres`

## 1. Contexte et décisions produit

Avant ce sous-lot, le périmètre d'un utilisateur était porté par
`bridge_user_role.perimetre_type` / `perimetre_id` (Lot 1). Un user
pouvait techniquement avoir plusieurs périmètres en cumulant
plusieurs rôles, mais la sémantique « droit RBAC » et « périmètre
budgétaire » étaient mélangées dans la même table.

Décisions produit (figées avant Lot 4.1) :

| Décision | Choix retenu | Conséquence |
|----------|--------------|-------------|
| **D1** — Cible des affectations | STRUCTURE + CR + CR_SET dès le MVP | Modèle 3-types implémenté, contrainte SQL `ck_user_perimetres_cible_coherence` |
| **D2** — Anti-chaînage des délégations | strict (pour Lot 4.2) | Champ `delegation_id` ajouté dès maintenant (FK vers `delegations` à venir) |
| **D3** — Événements email | 8 (E1, E2, E3, E4, E5, E7, E8, E9 — E6 reporté Lot 6) | Hors périmètre 4.1 (sera traité au 4.3) |

## 2. Modèle de données

### 2.1 Nouvelle table `user_perimetres`

Migration : `1779200000080-AjoutUserPerimetres.ts`.

| Colonne | Type | Notes |
|---------|------|-------|
| `id` | bigint PK identity | |
| `fk_user` | bigint NOT NULL | FK `user(id)` ON DELETE CASCADE |
| `cible_type` | varchar(20) NOT NULL | `STRUCTURE` / `CR` / `CR_SET` (CHECK) |
| `cible_id` | bigint NULL | rempli pour STRUCTURE / CR |
| `cible_cr_ids` | bigint[] NULL | rempli pour CR_SET (≥ 2 éléments) |
| `origine` | varchar(30) NOT NULL | `PRINCIPAL` / `AFFECTATION` / `DELEGATION` |
| `delegation_id` | bigint NULL | FK `delegations(id)` à brancher au Lot 4.2 |
| `date_debut` | date NOT NULL | défaut SQL `CURRENT_DATE` |
| `date_fin` | date NULL | NULL = permanent |
| `actif` | boolean NOT NULL | défaut `true` (soft-delete via `false`) |
| `motif` | text NULL | |

### 2.2 Index

- `idx_user_perimetres_user_actif (fk_user, actif)`
- `idx_user_perimetres_cible (cible_type, cible_id)`
- `idx_user_perimetres_cr_set USING GIN (cible_cr_ids)`
- `uq_user_perimetres_actif (fk_user, cible_type, cible_id, origine) WHERE actif AND cible_type IN ('STRUCTURE','CR')`

### 2.3 Contrainte de cohérence

```sql
CHECK (
  ( cible_type IN ('STRUCTURE','CR')
    AND cible_id IS NOT NULL
    AND cible_cr_ids IS NULL )
  OR
  ( cible_type = 'CR_SET'
    AND cible_id IS NULL
    AND cible_cr_ids IS NOT NULL
    AND array_length(cible_cr_ids, 1) >= 2 )
)
```

### 2.4 Backfill

Lors de l'application de la migration, toutes les lignes
`bridge_user_role` actives avec `perimetre_type IN ('structure','centre_responsabilite')`
sont recopiées dans `user_perimetres` avec :

- `cible_type = 'STRUCTURE'` ou `'CR'` selon le type d'origine
- `cible_id = perimetre_id`
- `origine = 'PRINCIPAL'`
- `date_debut = COALESCE(date_debut_validite, CURRENT_DATE)`
- `actif = est_actif`

Les rôles globaux (`perimetre_type='global'` ou NULL) ne génèrent
PAS de ligne — l'admin global est toujours détecté via le bridge.

> **Vérification post-migration** :
> `SELECT COUNT(*) FROM bridge_user_role bur
>     WHERE bur.perimetre_type IN ('structure','centre_responsabilite')
>       AND bur.perimetre_id IS NOT NULL AND bur.est_actif = true
>       AND NOT EXISTS (SELECT 1 FROM user_perimetres up WHERE up.fk_user = bur.fk_user)`
> doit retourner **0** (validé en local au 6 mai 2026).

## 3. Backend

### 3.1 Extension `PerimetreService`

Fichier : `src/budget/services/perimetre.service.ts`.

- Signature publique inchangée pour préserver le filtrage Q5
  (Lot 3.3) : `getCrAutorisesPourUser(userId)` retourne toujours
  `null` (admin global) ou `string[]`.
- Logique interne refactorée :
  - court-circuit `null` si `bridge_user_role.perimetre_type = 'global'`
  - sinon **union** entre `getPerimetreEffectif(userId)` (nouvelle
    source `user_perimetres`) et `crsViaBridgeUserRole(roles)`
    (lecture historique conservée le temps d'un cycle — dette
    tracée Lot 6).
- Nouvelle méthode publique `getPerimetreEffectif(userId, dateRef?)`
  qui parcourt `user_perimetres` actives à `dateRef` (par défaut
  aujourd'hui) :
  - STRUCTURE → BFS sur `dim_structure.fk_structure_parent`
  - CR → ajout direct (vérification `version_courante`)
  - CR_SET → ajout direct des `cible_cr_ids` (vérification en lot)
  - déduplication finale.

### 3.2 Service applicatif `UserPerimetreService`

Fichier : `src/users/services/user-perimetre.service.ts`.

3 méthodes publiques :

- `creer(userId, dto, auteurEmail)` — valide, inserte, audit
  `AFFECTATION_CREEE`.
- `retirer(userId, perimetreId, auteurEmail)` — soft-delete
  (`actif=false`), audit `AFFECTATION_RETIREE`.
- `lister(userId, options)` — lecture brute (filtres `actif`,
  `origine`, `dateRef`).

Validation métier (BadRequest 400 sinon) :

- Cohérence `cible_type` ↔ `cible_id` / `cible_cr_ids` (la
  contrainte SQL est doublée par une validation applicative qui
  donne un message d'erreur lisible).
- CR_SET : ≥ 2 CR distincts.
- Existence des cibles en base (`version_courante=true`,
  `est_actif=true`).
- `date_fin >= date_debut` si fournie.

### 3.3 API REST

| Méthode | URL | Permission | Description |
|---------|-----|-----------|-------------|
| `GET` | `/admin/users/:userId/perimetres` | `USER.LIRE` | Liste les affectations (filtres `actif`, `origine`, `dateRef`) |
| `POST` | `/admin/users/:userId/perimetres` | `USER.GERER` | Crée une affectation |
| `DELETE` | `/admin/users/:userId/perimetres/:id` | `USER.GERER` | Désactivation soft (audit) |
| `GET` | `/me/perimetres` | (auth) | Mes affectations actives à aujourd'hui |

Tous les endpoints sont sous le préfixe global `/api/v1`.

### 3.4 Audit

Deux nouveaux `type_action` ajoutés à l'enum `TypeAction` :
`AFFECTATION_CREEE`, `AFFECTATION_RETIREE`. Le payload JSON
contient `userId`, `cibleType`, `cibleId`, `cibleCrIds`, `origine`,
et les dates d'effet.

> **Note d'exploitation** : l'enum DB `ref_type_action_audit` doit
> être enrichi en parallèle (cf. paquet B fix B.3 — Lot 4.2 ajoutera
> les codes `AFFECTATION_*` à la table de référence).

## 6. Smoke test reproductible

1. `admin@miznas.local` se connecte → `/admin/affectations`.
2. Clic « Gérer » sur `lecteur@miznas.local` → ajoute :
   - 1 affectation STRUCTURE ciblant `CIV`
3. `lecteur@miznas.local` se reconnecte → header affiche
   « 1 périmètre ». Clic → modal liste l'affectation.
4. `/budget/saisie` → la grille affiche les CR de la sous-arbo CIV.
5. Admin ajoute une 2e affectation `CR_SET = [BR_BFA, BR_TGO]`
   pour le lecteur → header passe à « 2 périmètres ». Saisie voit
   l'union (CIV + 2 CR explicites).
6. Admin retire la 2e → lecteur revient à « 1 périmètre ».

## 7. Dette résiduelle (à traiter Lot 6 ou avant)

- **Coexistence `bridge_user_role` ↔ `user_perimetres`** : `getCrAutorisesPourUser`
  fait l'union des deux pour la rétrocompatibilité. Une fois la
  bascule complète stabilisée (tous les déploiements ont exécuté la
  migration), retirer la lecture `bridge_user_role.perimetre_type`
  dans `crsViaBridgeUserRole`.
- **Codes `ref_type_action_audit`** : ajouter `AFFECTATION_CREEE` /
  `AFFECTATION_RETIREE` dans la table de référence (migration au
  Lot 4.2 lors de l'ajout des autres types audit).
- **FK `delegation_id`** : actuellement `bigint NULL` sans
  contrainte référentielle. La FK vers `delegations(id)` sera
  posée par la migration du Lot 4.2.
- **`pg-mem` et `WITH RECURSIVE`** : la descente d'arbre STRUCTURE
  reste en BFS itératif côté JS — performance acceptable jusqu'à
  ~10 000 structures. Bascule conditionnelle sur `WITH RECURSIVE`
  prévue au Lot 6.
